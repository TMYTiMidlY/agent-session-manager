# ADR 0004：恢复侧——忠实迁移的保真度与破坏性操作安全

## 状态

已接受为**方向**；实现在 GitHub issue 跟踪。是 [ADR 0001](0001-scope-archive-and-restore.md) 恢复侧的契约。完整的跨机迁移操作手册不放本仓库（属外部运维知识）。

## 背景

"恢复" = 让会话在另一台机器上还能被 CLI `--resume` 续聊，来源 = restic 快照 ∪ 另一台机器。它与归档是**两种不同的保真度契约**（见 ADR 0001 的"归档 ≠ 恢复"）。

## 决策

恢复侧遵循以下理念（三家 agent 的磁盘形态不同，但原则一致）：

- **搬的是原生会话状态，不是报告**：Copilot = 会话目录 + `session-store.db` 行；Claude / Codex = 原生 `*.jsonl`。报告有损、不可 resume，不作恢复输入。
- **改功能字段，不动历史正文**：跨机换 home / cwd 时只按前缀改写**功能字段**（Copilot 的 cwd 藏在 `workspace.yaml`、`events.jsonl` 的 `session.start.context`、中心 DB 三处，外加 `git_root`）；而 events 正文 / todo / 会话名里出现的旧路径是**历史事实，保留才忠实**——绝不对整目录做"旧→新"全文替换。
- **"名字/时间"与中心索引各有来源**：Copilot 的名字来自 `workspace.yaml`、时间与排序来自**文件 mtime**、中心 DB 只服务 `/chronicle` 搜索；合并 DB 时按行合并、绝不整库覆盖目标已有会话。
- **明确的取舍要显式记录**：`rewind-snapshots/` 体积最大、只服务 `/rewind`、与 resume 无关，可按策略排除并记为"已确认排除"，不算漏传。
- **删除是危险操作，必须分层、可审计、可回滚**：迁移会逐字带过来云端 `mc_*` 挂钩，使源 / 目标共享同一个云端 task——批量删除务必只删本地（`--local-only`），否则会连累云端与其它设备。删源走 **隔离 → 回收站 → 独立只读核验 → 永久删** 四层，并配一份不可变清单做集合 / 目录 / 元数据 / 路径 / 行为 / 环境的六层验收；"命令退出码 0"不等于"删对了且没误伤"。

## 后果

恢复面对的是**易篡改、跨机会漂移、且删错不可逆**的原生状态，所以它的边界比归档严得多：宁可显式记录取舍、分层核验，也不做"一把梭"的整体替换或删除。
